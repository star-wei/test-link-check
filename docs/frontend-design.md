## 一、项目背景与核心诉求

### 1.1 背景
- 后端对接报价单系统，自动同步订单数据。
- 算法层已完成：工厂分配 → 拆件 → 装箱方案计算。
- 前端核心任务：**可视化展示装箱方案**、**支持人工微调**、**导出 PDF**。

### 1.2 已有人工范例分析
- **Excel 范例文件**：`装箱方案2025-1245-范例.xlsx`
- **排版特点**：
  - 按 **箱号（C1~C30）** 分组，每箱一个区块。
  - 每个区块列出该箱内所有组件的：编号、名称、成品尺寸（宽×高×厚）、数量、面积、包装尺寸备注。
  - 同箱组件共用左侧箱号列（合并单元格效果）。
  - 每个区块底部有汇总行：总数量、总面积。
- **箱型数据库**：`box_packing_solver.py` 中定义了 30 种箱型（C1~C30），包含：品牌、长、宽、高、体积(m³)、净重(kg)、毛重(kg)、备注。

### 1.3 重点关注
1. **直观展示"一个箱子里装了哪几个组件"**
2. **展示各工厂的装箱汇总**
3. **快速发现"超重"或"超尺寸"异常**

## 二、前端技术栈建议

### 2.1 推荐方案：React 18 + TypeScript + Ant Design 5.x

| 维度 | 选型 | 原因 |
|------|------|------|
| 框架 | **React 18** | 生态成熟、组件化开发、PDF 生成库丰富 |
| 语言 | **TypeScript** | 装箱数据字段多，强类型可减少接口对接错误 |
| UI 组件库 | **Ant Design 5.x** | 表格（Table）、卡片（Card）、标签（Tag）、统计（Statistic）等企业级组件成熟，适合 B 端后台 |
| 状态管理 | **Zustand** 或 **React Query (TanStack Query)** | 订单/方案数据以服务端状态为主，React Query 缓存+轮询体验好；局部交互状态用 Zustand 轻量 |
| 路由 | **React Router v6** | 四页结构清晰，嵌套路由或独立路由均可 |
| 图表 | **ECharts / Ant Design Charts** | 体积利用率、工厂箱数分布等可视化 |
| PDF 生成 | **前端：html2canvas + jsPDF / html-to-image + PDF-lib**；**后端：Node.js + Puppeteer + PDF-lib** | 详见第四章 |

### 2.2 备选方案
- **Vue 3 + Element Plus**：团队若熟悉 Vue 可替换，但 React 在复杂表格交互和 PDF 生态上略胜一筹。

### 3.2 装箱方案预览页（/orders/:id/packing）

**功能**：按工厂分组，展示每个箱子里的组件清单、尺寸、重量、利用率。这是系统的核心页面。

#### 3.2.1 页面布局

```
┌─────────────────────────────────────────────────────────────┐
│  面包屑：订单列表 > DW2025-1245 > 装箱方案                    │
├─────────────────────────────────────────────────────────────┤
│  订单概览卡（客户、总组件数、总箱数、总体积、总净重）          │
├─────────────────────────────────────────────────────────────┤
│  工厂筛选 Tabs：全部 | 绿盾 | 凯研 | 金牌车库门 | 极筑 | 普瑞斯 │
├─────────────────────────────────────────────────────────────┤
│  左侧：箱型导航树（按工厂→箱号）                              │
│  右侧：方案详情区（当前选中箱的组件清单 + 3D/2D 示意）         │
└─────────────────────────────────────────────────────────────┘
```

#### 3.2.2 左侧箱型导航树

- 按 **工厂** 分组折叠面板（Collapse / Tree）。
- 每个工厂下挂 **箱号节点**（C1, C2, C3...）。
- 箱号节点展示微标：
  - 体积利用率进度条（小）
  - 异常红点（超重 / 超尺寸 / 未匹配）
- 点击箱号，右侧滚动定位到对应区块（或右侧展示该箱详情）。

#### 3.2.3 右侧方案详情区

**设计思路**：参考 Excel 的"按箱分组区块"排版，但用 Web 表格增强可读性。

每个箱号一个 **Card 卡片**，卡片头部展示箱型元信息，卡片内是组件表格。

**Card 头部信息**：

```
┌────────────────────────────────────────────────────────────────────────┐
│  📦 C18  │  品牌：凯研  │  箱子尺寸：3660×760×1540 mm                │
│          │  体积：4.28 m³  │  净重上限：1121 kg  │  毛重上限：1228 kg │
│          │  体积利用率：78.5%  [进度条]  │  状态：✅ 正常            │
└────────────────────────────────────────────────────────────────────────┘
```

**Card 内表格字段**（参考 Excel 范例）：

| 字段 | 说明 | 示例 |
|------|------|------|
| 组件编号 | 拆件后的唯一编号 | W2-P1、D10-P2 |
| 组件名称 | 产品名称 | 110固定窗、Garage Door |
| 规格/系列 | 系列型号 | Z80、E70、DW105S |
| 成品尺寸 (mm) | 宽 × 高 × 厚 | 1067 × 2438 × 110 |
| 包装尺寸 (mm) | 宽 × 高 × 厚（含缓冲） | 1127 × 2528 × 140 |
| 数量 | 件数 | 1 |
| 单件面积 (㎡) | 自动计算 | 2.60 |
| 单件净重 (kg) | 算法估算或手动录入 | 142.5 |
| 备注 | 叠放上层、斜撑铁架等 | 叠放上层 |

**汇总行**（表格底部）：
- 总数量：Σ 数量
- 总面积：Σ 单件面积
- 总净重：Σ 单件净重
- 体积利用率：组件总体积 / 箱子体积 × 100%

#### 3.2.4 异常高亮规则

| 异常类型 | 判定规则 | 前端展示 |
|----------|----------|----------|
| **超重** | 组件总净重 > 箱子净重 × 安全系数（默认 0.9） | 箱号 Card 头部标红 + `Tag` 显示"超重"；表格总净重单元格背景变红 |
| **超尺寸** | 任一组件包装尺寸 > 箱子内径尺寸 | 对应组件行标红 + Tooltip 提示"包装尺寸超过箱型内径" |
| **利用率过低** | 体积利用率 < 30% | 黄色警告 Tag "利用率偏低" |
| **利用率过高** | 体积利用率 > 95% | 橙色警告 Tag "空间过紧" |
| **无匹配箱型** | 算法未找到合适箱型 | 组件行显示红色 Tag "无匹配箱型" |

#### 3.2.5 箱内组件可视化（解决"一个箱子装哪几个组件"）

在每张 Card 的表格上方，增加一个 **"箱内布局示意图"** 小区域：

- **2D 俯视图/侧视图**：用不同颜色矩形块表示每个组件在箱子内的占位（长×宽）。
- 若算法层能输出具体摆放坐标，则按坐标绘制；若不能，则按"组件列表"形式展示缩略色块图例。
- 每个色块 hover 显示组件编号和尺寸。
- 右侧附 **图例**：组件编号 → 颜色对应关系。

> MVP 阶段可用简化版：每个 Card 顶部放一个 `Space` 组件，用不同颜色 `Tag` 横向排列展示箱内组件编号，直观表达"这些组件都在这一箱"。

### 3.3 方案对比/调优页（/orders/:id/tuning）

**功能**：允许人工微调箱型分配，对比调整前后的差异。

#### 3.3.1 页面布局

```
┌─────────────────────────────────────────────────────────────┐
│  面包屑：订单列表 > DW2025-1245 > 方案调优                    │
├─────────────────────────────────────────────────────────────┤
│  模式切换：单箱调优  /  全局对比                              │
├─────────────────────────────────────────────────────────────┤
│  左侧：组件清单树（按订单→产品→拆件）                         │
│  右侧：调优操作区                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 3.3.2 单箱调优模式

- 右侧上方：当前方案的所有箱号卡片（与预览页类似，但可交互）。
- 操作：
  - **拖拽组件**：将组件从箱 A 拖到箱 B（HTML5 Drag & Drop）。
  - **更换箱型**：下拉选择其他可用箱型，实时重新计算利用率、重量、尺寸是否合规。
  - **拆箱/合箱**：支持将一箱拆分为两箱，或将两箱合并。
  - **撤销/重做**：操作历史栈。

- 每次操作后，右侧下方 **实时刷新差异对比卡片**：
  - 调整前：箱数、总体积、总净重、平均利用率
  - 调整后：同上
  - 差异：↑↓ 箭头 + 数值变化

#### 3.3.3 全局对比模式

- 左右分栏：
  - 左侧：算法原始方案（只读）
  - 右侧：人工调优后方案（可编辑）
- 底部：汇总对比表格

| 指标 | 原始方案 | 调优后方案 | 差异 |
|------|----------|------------|------|
| 总箱数 | 12 | 11 | -1 |
| 总体积 | 35.2 m³ | 33.8 m³ | -1.4 m³ |
| 总净重 | 4,850 kg | 4,920 kg | +70 kg |
| 平均利用率 | 72% | 78% | +6% |
| 异常箱数 | 2 | 0 | -2 |

#### 3.3.4 调优约束提示

当用户操作导致异常时，弹窗或 Tooltip 提示：
- "当前组件总净重 1150kg，超过 C18 净重上限 1121kg，请更换更大箱型或拆箱。"
- "包装尺寸 3200×1600×1550 超过 C30 内径，无法装入。"

## 四、PDF 导出方案（前端生成 vs 后端生成）

### 4.1 方案对比

| 维度 | 前端生成 | 后端生成 |
|------|----------|----------|
| **实现方式** | React 页面 → html2canvas → jsPDF / PDF-lib | Node.js + Puppeteer/Playwright 渲染 HTML → PDF |
| **开发成本** | 低，直接复用现有 DOM 和 CSS | 中，需要维护一套 PDF 专用 HTML 模板 |
| **样式还原度** | 中，复杂表格分页、页眉页脚控制较弱 | 高，Puppeteer 支持完整 CSS 打印媒体查询 |
| **分页精度** | 弱，大表格跨页容易截断 | 强，可用 `@media print` 和 `page-break` 精确控制 |
| **性能** | 大数据量时浏览器卡顿 | 服务端异步生成，不影响前端体验 |
| **文件大小** | 中（图片型 PDF 较大） | 小（矢量文字型 PDF） |
| **离线可用性** | 好 | 依赖网络 |

### 4.2 推荐方案：混合策略

#### 阶段一（MVP）：前端生成
- 使用 `html-to-image`（比 html2canvas 更清晰）将预览页 DOM 转为高清图片，再用 `pdf-lib` 组装成 PDF。
- 优点：快速上线，无需后端额外开发。
- 缺点：PDF 为图片型，文件较大，文字不可选中。

#### 阶段二（稳定期）：后端生成
- 后端使用 **Puppeteer + Handlebars/EJS 模板** 渲染与前端一致的 HTML，导出为矢量 PDF。
- 前端 PDF 预览页作为"发送参数"的确认页面，点击导出后调用后端 API：`POST /api/orders/:id/packing/pdf`。
- 后端异步生成 PDF，通过 WebSocket 或轮询通知前端下载。

### 4.3 前端 PDF 实现细节（MVP）

```ts
// 伪代码
import { toPng } from 'html-to-image';
import { PDFDocument, PageSizes } from 'pdf-lib';

async function exportPDF(domRefs: HTMLElement[]) {
  const pdfDoc = await PDFDocument.create();
  const a4 = PageSizes.A4;

  for (const ref of domRefs) {
    const dataUrl = await toPng(ref, { pixelRatio: 2 });
    const img = await pdfDoc.embedPng(dataUrl);
    const page = pdfDoc.addPage(a4);
    const { width, height } = page.getSize();
    page.drawImage(img, { x: 0, y: 0, width, height });
  }

  const bytes = await pdfDoc.save();
  download(bytes, `装箱方案_${orderNo}.pdf`, 'application/pdf');
}
```

## 五、数据模型设计（前端视角）

### 5.1 核心类型定义

```typescript
// 箱型定义（来自 box_packing_solver.py）
interface BoxType {
  boxId: string;        // C1 ~ C30
  brand: string;        // 极筑、绿盾、车库门、折叠门、凯研、凯撒
  length: number;       // mm
  width: number;        // mm
  height: number;       // mm
  volume: number;       // m³
  netWeight: number;    // kg
  grossWeight: number;  // kg
  remark: string;
}

// 组件（拆件后）
interface Component {
  componentId: string;      // W2-P1、D10-P2
  name: string;             // 110固定窗、Garage Door
  series: string;           // Z80、E70、DW105S
  finishedSize: Size3D;     // 成品尺寸
  packedSize: Size3D;       // 包装尺寸（含缓冲）
  quantity: number;
  unitArea: number;         // ㎡
  netWeight: number;        // kg
  remark?: string;
}

interface Size3D {
  width: number;
  height: number;
  depth: number;
}

// 装箱单元（一个箱子）
interface PackingBox {
  boxId: string;            // 实际使用的箱号 C18
  boxType: BoxType;         // 箱型元信息
  factory: string;          // 所属工厂
  components: Component[];
  totalQuantity: number;
  totalArea: number;
  totalNetWeight: number;
  volumeUtilization: number; // %
  weightOk: boolean;
  sizeOk: boolean;
  warnings: WarningType[];
}

type WarningType = 'OVERWEIGHT' | 'OVERSIZE' | 'LOW_UTILIZATION' | 'TIGHT_SPACE' | 'NO_MATCH';

// 订单级装箱方案
interface PackingPlan {
  orderNo: string;
  buyer: string;
  boxes: PackingBox[];
  factorySummary: FactorySummary[];
}

interface FactorySummary {
  factory: string;
  boxCount: number;
  totalVolume: number;
  totalNetWeight: number;
  avgUtilization: number;
}
```

## 六、关键交互与视觉设计

### 6.1 色彩规范（异常状态）

| 状态 | 颜色 | 用途 |
|------|------|------|
| 正常 | `#52c41a`（绿） | 合规箱号、正常数据 |
| 超重 | `#f5222d`（红） | 总净重超过箱型上限 |
| 超尺寸 | `#f5222d`（红） | 组件包装尺寸超过箱型内径 |
| 利用率过低 | `#faad14`（橙黄） | < 30% |
| 利用率过高 | `#fa8c16`（橙） | > 95% |
| 信息提示 | `#1890ff`（蓝） | 箱号、品牌标识 |

### 6.2 表格展示形式（参考 Excel）

Web 端对 Excel 排版的优化：
- **取消合并单元格**，改用 Card 分组，每箱独立一张 Card，避免表格跨行合并带来的可读性问题。
- **箱号作为 Card 标题**，左侧加工厂色条（不同工厂不同颜色）。
- **表格内数据右对齐**：尺寸、数量、面积、重量等数字类字段右对齐，便于快速扫描。
- **汇总行加粗 + 背景色区分**：浅灰色背景，字体加粗。

### 6.3 工厂汇总展示

在装箱方案预览页顶部，增加 **FactorySummaryBar** 横向卡片条：

```

│  3 箱   │ │  8 箱   │ │  6 箱   │ │  2 箱   │
```

点击汇总卡片可快速筛选到对应工厂的箱号列表。

## 七、接口对接建议（前端 ↔ 后端）

### 7.1 关键 API

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/orders` | GET | 获取订单列表 |
| `/api/orders/:id/packing` | GET | 获取装箱方案详情 |
| `/api/orders/:id/packing` | PUT | 保存人工调优后的方案 |
| `/api/orders/:id/packing/pdf` | POST | 触发后端 PDF 生成（阶段二） |
| `/api/box-types` | GET | 获取箱型数据库（C1~C30） |

### 7.2 调优页数据流

1. 进入调优页时，前端 GET 原始方案数据。
2. 用户在本地进行拖拽、换箱等操作（不实时调用接口）。
3. 点击"保存调优"时，前端本地计算校验规则，通过后 PUT 到后端。
4. 后端返回保存后的最新方案，前端刷新页面状态。

## 十、总结

本设计文档基于用户提供的 **Excel 装箱范例** 和 **箱型算法数据**，为门窗企业装箱方案系统规划了完整的前端架构：

1. **四页结构**覆盖从订单浏览、方案预览、人工调优到 PDF 导出的完整流程。
2. **React + Ant Design** 技术栈兼顾开发效率与企业级体验。
3. **按箱号 Card 分组 + 异常色标 + 工厂汇总条** 的设计，直观回答了三个核心问题：
   - 一个箱子里装了哪几个组件 → **每箱独立 Card + 组件表格 + 色块图例**
   - 各工厂装箱汇总 → **顶部 FactorySummaryBar + 左侧工厂导航树**
   - 超重/超尺寸异常 → **红橙黄三色 Tag + 表格单元格背景高亮 + 实时校验提示**
4. **PDF 采用"前端 MVP → 后端正式"的渐进策略**，平衡上线速度与长期体验。
