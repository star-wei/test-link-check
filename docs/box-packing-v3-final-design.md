## 一、业务目标

- 从报价单自动读取订单数据

- 按工厂批量优化装箱方案
- **算法推荐 + 人工确认/微调 + 记录修正原因用于后续 ML 训练**

## 三、混合拆件策略

### 3.1 自动拆件（标准品）

| 产品类型 | 拆件规则 | 说明 |
|---|---|---|
| **固定窗** | 宽度按 `1143mm` 等分扇数，向上取整 | 边扇可能加 30mm 框架余量 |
| **推拉门** | 每扇默认 ≤ `1300mm`，超过自动加扇数 | 按轨位拆分 |
| **入户门** | **不拆**，整扇一个组件 | 含门框包装 |
| **车库门** | 按标准门板单元拆分 | 从规格字段读取 |
| **折叠门** | 按折叠扇数量拆分 | 从规格字段读取 |
| **定制产品** | 触发人工拆件 | 铝木/幕墙等复杂结构 |

## 四、Padding 规则

**统一公式**：
- `包装宽 = 成品宽 + 60mm`
- `包装高 = 成品高 + 60mm`
- `包装厚 = 成品厚 + 30mm`

> 后续可扩展为按易碎度/运输距离动态调整。

## 五、重量计算

```python
组件重量(kg) = (成品宽 × 成品高 / 1,000,000) × 玻璃系数
```

| 玻璃类型 | 系数 |
|---|---|
| 双玻 | 50 kg/㎡ |
| 三玻 | 75 kg/㎡ |

## 六、箱型规则（动态定制）

**❌ 不再使用固定 C1-C30 箱型池**

### 6.1 单箱尺寸约束（硬约束）

| 维度 | 约束 |
|---|---|
| **宽 (W)** | `500mm ≤ W ≤ 1200mm` |
| **高 (H)** | `H ≤ 2500mm` |
| **长 (L)** | 基本不限（实践中通常 ≤ 7000mm） |
| **总重量** | `≤ 1500 kg` |

### 6.2 多件同箱的摆放方式

**只允许并排立放（方案 B）**：

```
┌─────────────────────────────────────────────┐
│  │ 组件1 │    │ 组件2 │    │ 组件3 │        │
│  │       │    │       │    │       │        │
│  └───────┘    └───────┘    └───────┘        │
│     并排站立，宽度方向累加                    │
└─────────────────────────────────────────────┘
```

**数学表达**：
- 先把每个组件旋转到最优姿态（枚举 6 种旋转）
- 然后沿 **宽度方向** 并排摆放
- 箱子的尺寸 =：
  - `箱宽 = 所有组件的包装宽度之和 + padding`
  - `箱高 = 所有组件的包装高度最大值 + padding`
  - `箱长 = 所有组件的包装长度最大值 + padding`

**关键规则**：
- 排完后必须满足 `500 ≤ 箱宽 ≤ 1200`
- 如果不满足，尝试旋转组件 90° 再排
- 如果还是不满足，说明这些组件不能放在同一箱内

### 6.3 为什么只允许并排立放？

业务约束：**不允许两个组件沿长边方向"串成一条线"排列**。因此：
- ❌ 沿长度方向首尾相接
- ✅ 并排站立（宽度累加）
- ✅ 上下堆叠（高度累加，作为备选方案）

## 七、批量装箱优化算法

### 7.1 输入

按工厂聚合后的组件池：
```python
{
  "凯研工厂": [W1-P1, W1-P2, W2-P1, ...],
  "绿盾工厂": [D5-P1, D5-P2, ...]
}
```

### 7.2 硬约束

1. `500 ≤ 箱宽 ≤ 1200`
2. `箱高 ≤ 2500`
3. `单箱总重 ≤ 1500 kg`
4. 所有组件必须能**并排立放**入箱（旋转后宽度之和 ≤ 箱宽上限）

### 7.3 优化目标（多目标评分）

| 目标 | 评分项 | 权重 |
|---|---|---|
| **A. 箱数最少** | 每箱固定加分 | 高 |
| **B. 体积利用率最高** | `util_score = 100 - abs(util - 80)` | 中 |
| **C. 重量接近 1500kg** | `weight_score = (total_weight / 1500) * 50` | 中 |
| **D. 同系列不散装** | 同一 product_id 的组件在同一箱时加分 | 高 |

**综合评分**：
```python
score = util_score * 0.3 + weight_score * 0.3 + group_bonus * 0.4 - penalty
```

### 7.4 算法流程

```
1. 按工厂聚合所有组件
2. 对每个工厂的组件池：
   a. 按 product_id 分组，同系列组件优先尝试合并为一箱
   b. 对每组组件：
      - 枚举每个组件的 6 种旋转方式
      - 尝试不同排列组合，找到满足箱宽 500-1200 的最优摆法
      - 计算该摆法下的箱高、箱长、总重
      - 选择评分最高的摆法
   c. 若同系列组件无法合并（太重或太宽），拆分为 2+ 箱
   d. 跨 series 的剩余组件，按体积从大到小尝试放入已有箱子
   e. 无法放入则开启新箱
3. 输出每个工厂的装箱明细
```

## 九、人工微调与学习机制

### 9.1 支持的人工操作

- 拖拽组件换箱
- 拆箱 / 合箱
- 更换箱子尺寸（调整长宽高）
- 旋转单个组件
- 调整 padding 值

### 9.2 修正原因记录字段

```json
{
  "adjustment_type": "split_box",
  "original_solution_id": "sol-123",
  "component_ids": ["W1-P1", "W1-P2"],
  "reason_category": "weight_limit",
  "reason_detail": "客户要求单箱不超过 1200kg",
  "operator": "张三",
  "timestamp": "2026-04-14T10:00:00Z"
}
```

### 9.3 原因分类（reason_category）

- `box_size` — 箱型不合适
- `weight_limit` — 重量限制
- `logistics` — 物流/运输限制
- `customer_request` — 客户指定
- `series_grouping` — 系列分组需求
- `other` — 其他

### 9.4 学习路径

**阶段 1（当前）**：记录 + 统计分析  
**阶段 2**：基于统计调整评分权重  
**阶段 3**：用历史数据训练监督学习模型，预测最优箱型和组件分组
